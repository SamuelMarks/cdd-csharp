# Publishing Generated Client Libraries

This document outlines how to publish the **C# Client Libraries** generated by `Cdd.OpenApi.Cli` and how to keep them automatically synchronized with the upstream OpenAPI specification.

## 1. Structuring the Client Library

When you run `cdd-openapi generate-code spec.json ./ClientDir`, it produces C# source files (`IApi.cs`, `Models/*.cs`).

To make this a publishable library:
1. Initialize a class library in that directory: `dotnet new classlib -n MyGeneratedClient -o ./ClientDir`
2. Add necessary dependencies (e.g., ASP.NET Core MVC abstractions if needed): `dotnet add ./ClientDir package Microsoft.AspNetCore.Mvc.Core`
3. Generate the code into the library folder.

## 2. Publishing the Client to NuGet

Once the code is generated, the process of publishing is identical to standard .NET libraries:

```bash
# Compile and pack the generated client
dotnet pack ./ClientDir/MyGeneratedClient.csproj -c Release -p:PackageVersion=1.0.0

# Publish it to your registry (NuGet.org or an internal feed like GitHub Packages/Azure Artifacts)
dotnet nuget push ./ClientDir/bin/Release/*.nupkg -k $NUGET_API_KEY -s https://api.nuget.org/v3/index.json
```

## 3. Automating Client Synchronization (GitHub Actions Cron Job)

The true power of an OpenAPI generator is keeping your client SDKs perfectly in sync with the upstream API. You can configure a GitHub Action to run periodically (e.g., daily), fetch the latest OpenAPI spec from the server, generate the C# code, and automatically publish a new NuGet package if there are changes.

Create a file at `.github/workflows/sync-client.yml` in your client's repository:

```yaml
name: Sync & Publish OpenAPI Client

on:
  schedule:
    # Run at 00:00 every day
    - cron: '0 0 * * *'
  workflow_dispatch: # Allows manual triggering

jobs:
  sync-and-publish:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Repository
        uses: actions/checkout@v3

      - name: Setup .NET
        uses: actions/setup-dotnet@v3
        with:
          dotnet-version: '10.0.x'

      - name: Install CDD OpenAPI CLI
        # Assuming you published the CLI tool globally
        run: dotnet tool install -g Cdd.OpenApi.Cli

      - name: Fetch Latest OpenAPI Spec
        run: curl -sSL https://api.example.com/openapi.json -o latest-spec.json

      - name: Generate Code
        # This overwrites the existing C# files with the new structure
        run: cdd-openapi generate-code latest-spec.json src/MyGeneratedClient

      - name: Check for Changes
        id: check_changes
        run: |
          if [[ -z $(git status -s src/MyGeneratedClient) ]]; then
            echo "No changes in the API spec."
            echo "changed=false" >> $GITHUB_OUTPUT
          else
            echo "API has changed. Committing and publishing."
            echo "changed=true" >> $GITHUB_OUTPUT
          fi

      - name: Bump Version & Pack (If Changed)
        if: steps.check_changes.outputs.changed == 'true'
        run: |
          # Use a date-based versioning scheme (e.g., 2026.2.27) or proper semantic versioning
          NEW_VERSION="$(date +'%Y.%m.%d').${{ github.run_number }}"
          
          # Pack the new client
          dotnet pack src/MyGeneratedClient/MyGeneratedClient.csproj -c Release -p:PackageVersion=$NEW_VERSION
          
          # Push to NuGet
          dotnet nuget push src/MyGeneratedClient/bin/Release/*.nupkg -k ${{ secrets.NUGET_API_KEY }} -s https://api.nuget.org/v3/index.json

      - name: Commit & Push Updates (If Changed)
        if: steps.check_changes.outputs.changed == 'true'
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add src/MyGeneratedClient latest-spec.json
          git commit -m "Auto-update generated client from upstream OpenAPI spec"
          git push
```

### How it Works:
1. **Cron Schedule**: It wakes up daily to check the upstream server.
2. **Fetch & Generate**: It pulls down `openapi.json` and runs `cdd-openapi generate-code`.
3. **Diff Detection**: It uses `git status` to see if the generated C# models or routes actually changed.
4. **Publish**: If changes exist, it builds a new version of the library (using a date-based version or build ID), pushes it to NuGet, and commits the generated files back to the repository. If the API hasn't changed, it exits quietly without cluttering the NuGet feed.
